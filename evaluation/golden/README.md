# Golden Set — writeup2md OCR evaluation

## Composition

45 hand-verified visual-block samples with known ground-truth text. Each
sample is a PIL-rendered screenshot of real code/config/log content (not a
renamed text file). The ground-truth text is recorded both in
`manifest.jsonl` under `gold_verbatim` and in `expected/<sample_id>.txt`.

## Coverage

- **Visual types**: code (23), configuration (6), http (5), terminal (3),
  diff (3), log (3), traceback (2).
- **Languages**: python (18), bash (7), http (4), diff (3), log (3), yaml (2),
  ini (2), json (1), toml (1), javascript (1), go (1), rust (1), java (1).
- **Themes**: light (43), dark (2).
- **Edge cases**: editor line numbers, cropped content, low-resolution image,
  punctuation-heavy, URLs, hashes, IP addresses, quoted strings with escapes,
  indentation-sensitive Python, merge conflict markers, multi-header HTTP.

## Size shortfall

Spec target was ≥100 samples or ≥40 with honest shortfall. This set has 45 —
above the 40 minimum. Growing to 100+ is planned for a future round by
extracting real visual blocks from the `test_samples/` PDF corpus (TASK_14)
and verifying their ground truth manually.

## Regenerating the fixtures

The fixtures are generated by `evaluation/golden/_generate.py` (if present) or
by the inline generator in the TASK_09 task work. Each PNG is deterministic
given its source lines. Do NOT edit the PNGs directly; edit the source
definitions and regenerate so `gold_verbatim` stays in sync with the image.

## Usage

```bash
writeup2md evaluate-ocr evaluation/golden/ --backend rapid --output reports/golden-eval/
```

This produces:
- `reports/golden-eval/results.jsonl` — per-sample metrics
- `reports/golden-eval/summary.json` — aggregate metrics
- `reports/golden-eval/by_visual_type.json` — metrics grouped by visual_type
- `reports/GOLDEN_SET_METRICS.json` — copy of summary
- `reports/GOLDEN_SET_METRICS.md` — human-readable summary
