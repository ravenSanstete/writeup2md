## Design decisions

- **PyMuPDF (fitz) instead of Docling for v1.** `docs/01_ARCHITECTURE.md` suggested Docling for reading-order candidates, but PyMuPDF's `get_text("blocks", sort=True)` already returns blocks in reading order with bounding boxes and block-type metadata. Using PyMuPDF alone keeps the dependency surface smaller and the resource behavior easier to bound. Docling can be added later behind the same adapter interface.
- **Code-like detection is heuristic and conservative.** It looks for HTTP request lines, diff markers, shell prompts, and Python indentation/keywords. False positives are possible but acceptable because the renderer treats `native_code` blocks as fenced code blocks with optional language. Importantly, we never *repair* code — we just emit what we extracted.
- **Heading detection is also heuristic** (short single-line text without sentence-ending punctuation). This is intentionally simple; rich PDFs with proper heading styles would benefit from font-size analysis which we defer to a future improvement.
- **Scanned-page threshold** is conservative: a page must have very few native-text characters AND low text density relative to page area. Real scanned PDFs trigger it; native-text PDFs do not.
- **Embedded image extraction** is best-effort: images that fail to extract (CMYK conversion errors, missing xrefs) are silently skipped with a warning. We do not crash the pipeline.
- **No PaddleOCR-VL in this task** — visual blocks are left as `review_required` for TASK_04 to resolve.
- **Resource behavior:** one page is loaded at a time via `pdf_doc.load_page(i)`, the rendered PNG bytes are written to disk immediately, and the Pixmap is released. We do not retain a list of full-resolution page images.

## Known limitations

- The PDF fixture generated by `fitz.insert_text` puts the heading "Exploitation" in the same text block as the preceding code, so the rendered Markdown shows "Exploitation" inside the code fence. This is a fixture artifact, not an adapter bug — real PDFs with proper block structure do not exhibit this. The adapter preserves whatever block structure PyMuPDF reports.
- Font-size-based heading-level detection is not yet implemented; all headings default to level 2.
- No PDF OCR text-layer reuse yet. PyMuPDF's `get_text("blocks")` returns whatever text layer exists, including OCR text layers, so this is partially covered, but we do not explicitly detect "this PDF has an OCR text layer" and skip rendering.
- No page-checkpoint resume for very large PDFs yet. TASK_05 adds document-level resume; page-level resume is a future improvement.
